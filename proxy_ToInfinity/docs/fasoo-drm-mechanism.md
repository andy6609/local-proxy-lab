# Fasoo DRM 동작 원리 — 왜 프록시엔 암호문이 잡히나

> 같은 파일이 **git bash로 열면 평문**, **크롬 업로드엔 암호문**으로 보이는 이유.
> 이 프로젝트에서 여러 번 헷갈렸던 지점이라 개념으로 정리. (실측 근거는 `ai-upload-fingerprints.md`)

## 1. 핵심 — 파일은 디스크에 "암호문 하나", 복호화는 **프로세스별로** 결정

DRM 보호 파일은 **디스크에 암호문 형태로 딱 하나** 저장돼 있다(`DRMONE ... Fasoo DRM ...`). 평문 사본이 따로 있는 게 아니다.

Fasoo DRM 에이전트는 백그라운드에서 **모든 파일 읽기(read)를 가로챈다(hook).** 어떤 프로세스가 그 파일을 읽으려 하면:

1. 디스크의 실제 바이트 = **암호문** (항상 이것 하나)
2. 에이전트가 판단: **"지금 읽는 프로세스가 허가된 앱인가?"**
   - **허가된 프로세스** → 읽는 순간 **자동 복호화**해서 건네줌 → 그 프로세스는 **평문**을 봄
   - **비허가 프로세스** → 복호화 안 함 → **암호문 그대로** 받음

즉 **"파일이 평문이냐 암호문이냐"는 파일 자체가 아니라 "누가 읽느냐"에 달려 있다.**

## 2. 그래서 같은 파일이 다르게 보인다 (실측)

| 읽는 프로세스 | DRM 판단 | 그 프로세스가 받는 것 |
|---|---|---|
| **git bash** (검증에 쓴 도구) | 허가됨 | **평문** 13699 B, `PK..[Content_Types].xml` |
| **크롬** (실제 업로더) | 비허가 | **암호문** 19856 B, `DRMONE ... Fasoo DRM` |

파일은 **하나**(디스크엔 암호문)인데, git bash는 복호화된 걸 보고 크롬은 암호문을 본다. 크롬이 업로드하면 그 **암호문**이 네트워크로 나가고, 그게 **프록시에 잡힌다.**

## 3. 예외(exception) 폴더 뉘앙스

- **거기서 "새로 만든" 파일** → 진짜 비-DRM → 모든 프로세스에 평문 (크롬 업로드도 평문 `PK`).
- **DRM 문서의 복사본/이동본**을 예외폴더에 둔 것 → **DRM 유지(지속보호)** → 여전히 암호문. `cp`로 만든 사본도 마찬가지.
  (그래서 예외폴더의 `EXCTEST_plaintext.docx`—DRM문서 복사본—가 크롬 업로드엔 암호문으로 나갔다.)

## 4. 프록시 관점에서의 결론 (가장 중요)

**프록시가 얻는 내용 = 업로드 앱(크롬)이 "읽도록 허가된" 내용.**

- **DRM 파일** → 프록시는 **메타데이터(파일명·크기·타입·엔드포인트·업로드방식)만** 확보. 본문은 암호문이라 못 읽음.
  게다가 **읽을 때마다 암호문이 달라짐**(랜덤 IV로 추정) → 내용 해시로 식별·중복판정도 불가.
- **비-DRM 파일** → 본문까지 평문으로 확보 (원본과 바이트 일치, `rootCA.crt` 업로드로 확인).

이게 곧 DLP 관점: **보호 안 된 평문 파일이 AI로 새면 프록시가 내용까지 잡고, DRM 보호 파일은 메타데이터만 보인다.**

## 5. ⚠️ 검증 함정 (반드시 기억)

**"이 파일이 평문인가"를 git bash 같은 Fasoo 허가 프로세스로 판단하면 안 된다.** 허가 프로세스는 DRM 파일도 복호화해서 평문으로 보여주기 때문. → **평문/DRM 판정은 반드시 "프록시가 잡은 것(= 업로드 앱이 실제 보낸 바이트)" 기준으로.**

## 6. 실험 설계 함의

| 목적 | 환경/파일 |
|---|---|
| 추출 파이프라인이 **본문을 정확히 뽑나** (평문 검증) | **진짜 비-DRM 파일**(웹 다운로드본, 새로 만든 비-DRM 타입 등) 또는 **무-DRM 환경(Hyper-V clean VM)** |
| **Fasoo DRM 동작** 연구(목적지별 정책, 프로세스 기반 등) | **DRM 깔린 호스트 필수** — Hyper-V(무DRM)는 연구 대상 자체를 없앰 |

즉 "제대로 실험"의 답은 하나가 아니라 **두 트랙**: DRM 발견은 호스트, 순수 추출검증은 clean 환경.
