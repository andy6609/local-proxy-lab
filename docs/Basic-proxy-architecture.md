# 프록시 구조 설명서 — HTTP 처리 (MITM 이전)

> **용도:** 발표/리뷰용. "우리 프록시가 어떤 구조로 흐르는가"를 설명 위주로 정리한다.
> **범위:** TLS MITM 으로 넘어가기 **전**까지 = HTTP 를 다루는 부분 (M1~M3).
> **관점:** 우리 학습 코드(`proxy_pure_*`)는 *단순 예시*로만 쓰고,
> 본문은 **모든 포워딩 프록시가 공통으로 갖는 구조** + **실제 고성능 프록시와 무엇이 다른가**에 무게를 둔다.

---

## 0. 한 줄 정의

**포워딩 프록시 = 클라이언트(브라우저)와 진짜 서버 사이에 끼어,
클라의 요청을 대신 서버에 전달하고 응답을 되돌려주는 중계자.**

```
브라우저  ──요청──▶  [프록시]  ──요청──▶  진짜 서버
브라우저  ◀──응답──  [프록시]  ◀──응답──  진짜 서버
```

프록시가 그 중간에서 **무엇을 보느냐**에 따라 단계가 갈린다:
- 바이트만 본다 (내용 모름) → **L4 릴레이** (M1)
- HTTP 를 해석한다 (메서드/URL/헤더/바디 경계) → **L7 파싱** (M2)
- HTTPS 터널을 연다 (암호는 통과) → **CONNECT 터널** (M3) → 이 자리가 MITM(M4) 진입점

---

## 1. 모든 프록시가 공통으로 갖는 골격 ★

단계(M1/M2/M3)가 달라도 **바깥 골격은 똑같다.** 가운데 "연결 처리"만 바뀐다.
이 골격은 어떤 프록시(nginx, Squid, mitmproxy …)든 개념적으로 동일하다.

### (a) 서버를 세운다 — listen 소켓

```
WSAStartup            # (Windows 전용) 소켓 라이브러리 초기화
  └ socket()          # 듣기용 소켓 1개 생성
     └ bind(:18080)   # "이 포트로 들어오는 연결을 내가 받는다"
        └ listen()    # 연결 대기 상태로 전환
           └ accept() 루프   # 연결 하나 들어올 때마다 새 소켓 반환
```

- `accept()` 가 반환하는 건 **연결 하나당 새 소켓**. listen 소켓은 계속 다음 연결을 받는다.
- **핵심 API:** `socket` / `bind` / `listen` / `accept` — 이거 없으면 서버 자체가 안 선다.

### (b) 연결 하나를 처리한다 — 동시성

```
accept() 로 받은 연결  →  처리 루틴(HandleClient) 실행
```

연결이 여러 개 동시에 들어오므로 **동시에 처리할 방법**이 필요하다.

| 모델 | 누가 쓰나 | 설명 |
|---|---|---|
| **연결당 스레드 (우리 코드)** | 학습용·소규모 | `accept` 할 때마다 스레드 1개 생성 후 분리(detach). blocking 소켓. |
| **이벤트 루프 (실제 고성능)** | nginx, Envoy, HAProxy | 스레드는 **CPU 코어 수만큼만**, non-blocking 소켓 + epoll/IOCP 로 수만 연결을 소수 스레드가 처리. |

> ⚠️ **여기가 우리 코드와 실제 프록시의 가장 큰 구조 차이다.** (자세히 §3)

### (c) 진짜 서버로 연결한다 — upstream

```
getaddrinfo(도메인, 포트)   # 도메인 이름 → IP 주소 목록 (DNS 조회)
  └ socket()               # 서버로 나갈 소켓 생성
     └ connect()           # 그 IP:포트로 실제 TCP 연결
        └ freeaddrinfo()   # 조회 결과 메모리 해제
```

- `getaddrinfo` 는 OS(Winsock) API. 한 도메인에 IP 가 여러 개일 수 있어 **목록(linked list)** 으로 돌려준다.
- **핵심 API:** `getaddrinfo` / `socket` / `connect` — 이거 없이는 목적지에 못 붙는다.

### (d) 데이터를 옮긴다 — recv / send

- `recv()`: 소켓에서 바이트를 받는다. 받을 게 없으면 **블록(대기)**.
- `send()`: 소켓으로 바이트를 보낸다. (한 번에 다 안 나갈 수 있어 **다 보낼 때까지 반복** = `sendAll`)
- `closesocket()` / `shutdown()`: 연결 정리. `shutdown(SD_SEND)` 은 "나는 더 안 보낸다" 신호 → 반대쪽이 `recv` 0 을 받아 같이 끝난다.

## 2. 가운데 "연결 처리"의 3가지 형태 (M1 → M2 → M3)

골격(§1)은 그대로 두고, **`HandleClient` 안에서 무엇을 하느냐**만 진화한다.

### M1 — Blind 릴레이 (L4, 내용 안 봄)

```
recv(첫 요청)  →  Host 헤더만 추출  →  upstream 연결
   →  첫 요청 전달
   →  ┌ [스레드]  Relay(클라 → 서버)   ┐
      └           Relay(서버 → 클라)   ┘  ← 양방향 동시
```

- **Relay = "받은 바이트를 그대로 반대편에 흘려보냄"**. HTTP 를 해석하지 않는다.
- **왜 스레드 2개?** 한 방향 `recv` 가 블록된 동안 반대 방향도 흘러야 하니까(full-duplex).
  스레드 1개면 한쪽 `recv` 에 갇혀 반대쪽이 멈춘다.
- 한계: 메시지 경계를 모른다 → body 를 정확히 못 떼어낸다(파일 추출 불가).

### M2 — HTTP 파싱 루프 (L7, **이번 발표의 핵심**)

```
반복 {
  ① 요청 헤드 파싱   readHead → 시작줄(method/URL/version) + 헤더들
  ② upstream 연결
  ③ 요청 헤드 + body 전달   (body 길이를 알고 그만큼 정확히)
  ④ 응답 헤드 파싱   readHead → status
  ⑤ 응답 헤드 + body 전달
  ⑥ keep-alive 면 반복, 아니면 종료
}
```

- **핵심 도구 `SockReader`:** TCP 는 스트림이라 `recv` 한 번에 메시지가 딱 안 떨어진다.
  - 헤더가 쪼개져 오거나(부분 수신) / body 까지 미리 읽히거나(over-read).
  - 그래서 "줄 단위로 읽기 / N바이트 정확히 읽기 / 잉여 보관" 하는 버퍼드 리더가 필요.
- **body 경계 판정(프레이밍)** — M2 의 진짜 학습 포인트:

  | 방식 | 판단 근거 |
  |---|---|
  | `Content-Length: N` | 정확히 N 바이트가 body |
  | `Transfer-Encoding: chunked` | 청크 크기(16진수)별로 끊어 읽음 |
  | 둘 다 없음 | 연결 종료까지가 body (응답만) |

- **이게 왜 중요한가:** body 가 어디서 끝나는지 알아야
  ① keep-alive(다음 요청 이어받기)가 되고
  ② **요청 body 를 정확히 추출**할 수 있다 → 나중에 **파일 업로드 탐지(M6~M7)의 직접 토대**.

### M3 — CONNECT 분기 (HTTPS, MITM 직전)

```
recv(첫 요청)  →  첫 줄의 메서드 판별
   ├─ "CONNECT example.com:443"  →  HandleConnect  (HTTPS 터널)
   └─ "GET / HTTP/1.1 ..."       →  HandleHttp     (평문 HTTP)
```

`HandleConnect` 흐름:
```
upstream(:443) TCP 연결
  → 브라우저에 "HTTP/1.1 200 Connection Established" 응답
  → ★ 터널이 열리는 순간 ★
  → 양방향 blind 릴레이 (암호 바이트를 그대로 통과 — 내용 못 봄)
```

- **CONNECT = L7 HTTP 메서드인데 "L4 TCP 터널을 뚫어줘"라는 요청.**
  프록시는 터널만 열어주고, 이후 브라우저가 서버와 직접 TLS 핸드셰이크 → 프록시는 암호문만 본다.
- ★ **바로 이 "200 직후" 자리가 M4(MITM) 진입점.**
  M4 는 blind 릴레이 대신 *프록시가 직접 양쪽 TLS 를 끊어* 복호화 → 평문 HTTP(M2 로직 재사용) → 재암호화.

## 3. 실제 고성능 프록시와 다른 점 (= 우리 구조의 한계) ★ 반드시 발표에 명시

우리 코드의 **로직**(HTTP 파싱·CONNECT·릴레이)은 실제 프록시와 동일하다.
하지만 다음 두 가지는 "학습/PoC 라서 단순화"한 부분이고, 실제 표준과 다르다.

### (A) I/O 동시성 모델 — 가장 큰 구조 차이

| | 우리 코드 (pure / v 시리즈) | 실제 고성능 프록시 |
|---|---|---|
| 소켓 | **blocking** | **non-blocking** |
| 동시성 | **연결당 스레드 1개** | **이벤트 루프** + 코어 수만큼 스레드 |
| 대기 방법 | `recv` 에서 블록 | epoll(Linux) / IOCP(Windows) / kqueue(BSD) |
| 한계 | 연결 1만 개 = 스레드 1만 개 → context switch 폭증 | 소수 스레드가 수만~수십만 연결 처리 |

> **왜 코어 수만큼?** 코어는 물리적 실행 단위. 스레드가 코어보다 많으면 번갈아 실행(context switch)하느라 오히려 느려진다.

### (B) 제품 하드닝 — pure 가 걷어낸 것들

발표에서 "우리 학습판은 본질만 남겼고, 실제 제품은 여기에 이것들이 더 붙는다"로 설명:

- **타임아웃** (느린/멈춘 연결 끊기)
- **헤더 크기 상한** (메모리 고갈 공격 방어)
- **Request Smuggling 방어** (`Content-Length` + `Transfer-Encoding` 동시 = 거부)
- **upstream 커넥션 풀** (매 요청 새 연결 대신 재사용)
- **HTTP/2, HTTP/3** 지원 (우리는 HTTP/1.1 텍스트만)
- **견고한 에러 처리 / 구조화 로깅**

> 참고: 우리 저장소의 `proxy_v1/2/3` 는 이 하드닝을 **일부 포함한** 버전이고,
> `proxy_pure_*` 는 그걸 **벗겨 흐름만 남긴** 학습판이다. (둘 다 I/O 모델은 blocking+스레드)

### 외부 레퍼런스 (동료와 비교해 볼 만한 실제 구조)

- **mitmproxy** — 우리 과제와 동일 use case(프록시 + TLS MITM 검사). addon/flow 구조.
- **nginx / Squid** — 포워드 프록시 + 이벤트 루프 아키텍처의 대표.

## 4. 발표 흐름 제안 (3단)

1. **도입 (M1):** 프록시 = 중계자. 양방향 릴레이가 왜 스레드 2개인가. (L4, 내용 안 봄)
2. **본론 (M2):** HTTP 를 *해석*하면 무엇이 가능해지나 — 메시지 경계/body 프레이밍 → 파일 추출의 토대. (L7)
3. **마무리 (M3 → 예고):** HTTPS 는 CONNECT 터널로 갈린다 → 암호라 못 본다 → **그래서 MITM(M4)이 필요**, 그 자리가 어디인가.
   + 우리 구조 vs 실제 고성능 프록시(이벤트 루프) 차이 1장.

## 부록 — 핵심 API 한눈에 (이거 없으면 안 돌아간다)

| 구간 | API | 역할 |
|---|---|---|
| 서버 수립 | `WSAStartup` `socket` `bind` `listen` `accept` | listen 소켓 + 연결 수락 |
| 동시성 | `std::thread` + `detach` (우리) | 연결당 처리 (실제: epoll/IOCP) |
| upstream | `getaddrinfo` `socket` `connect` `freeaddrinfo` | 도메인→IP, 진짜 서버 연결 |
| 데이터 | `recv` `send` `shutdown` `closesocket` | 바이트 송수신·정리 |
| (M2 추가) | `SockReader`(readLine/readExact/forwardExact/forwardChunked) | HTTP 파싱·body 프레이밍 |
| (M3 추가) | 첫 줄 메서드 판별 → CONNECT 분기 | HTTPS 터널 vs 평문 |
