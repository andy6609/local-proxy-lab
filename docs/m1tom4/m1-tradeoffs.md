# M1 트레이드오프 & 아쉬운 점

> M1(proxy_v1) 은 "양방향 릴레이 검증" 단계 산출물이다.
> 제품이 아니라 토대 — 한계를 알고 쓰는 게 목적.

---

## 실제로 위험한 것들 (제품이면 반드시 고쳐야 함)

### 1. `recv` 한 번으로 Host 파싱 → 깨질 수 있음

```cpp
int n = recv(clientSock, buffer, sizeof(buffer), 0);
ParseHost(buffer, ...);
```

TCP는 스트림이라 첫 `recv`에 헤더가 다 안 올 수 있다.
`"GET / HTTP/1.1\r\nHo"` 까지만 오면 `Host:`를 못 찾고 연결을 버린다.
→ **M2에서 해결 예정** (부분수신 누적 파싱).

### 2. 에러 처리가 로그만 찍고 끝

```cpp
shutdown(to, SD_SEND);  // 반환값 안 봄
// Relay 안에서 send 실패해도 조용히 넘어가는 경로 있음
```

제품은 모든 실패를 정책으로 다뤄야 한다 (재시도 / 로깅 / 메트릭).

### 3. 스레드 per 연결 → 확장성 한계

연결 1000개 = 스레드 2000개.
제품 프록시는 보통 epoll / IOCP 같은 이벤트 모델을 쓴다.
**단, 이 PoC 규모에선 용인 가능 — 보고서에 트레이드오프로 명시하면 됨.**

### 4. `buffer[16384]` 고정

헤더가 16KB 넘으면 처리 못 한다.
제품은 동적 버퍼나 스트리밍 방식.

---

## "아쉽지만 정상"인 이유

책임님 말: **"프록시는 AI 딸깍으로 쉽게 나온다. 중요한 건 바이너리 해석이다."**

회사가 평가하는 건 "프록시 코드 품질"이 아니라:
1. **왜 이렇게 동작하는지 설명할 수 있는가** (W10 발표 1순위)
2. **파일 업로드 식별/추출을 해냈는가** (M6~M9, 본 과제)

M1 코드는 "소켓/스레드/릴레이를 이해했다"는 증거로 충분하다.

---

## 우선순위 판단

지금 M1을 제품 수준으로 다듬는 데 시간 쓰지 말고, **M2~M5로 진도 빼고 M6~M9(파일 탐지)에 시간을 몰아라.**

최종 제출 직전(M9 근처)에:
- 에러 처리 / 자원 정리 한 번 훑고
- "이 트레이드오프는 PoC라 이렇게 했다"를 **보고서에 명시**

완벽한 코드보다 **"한계를 아는 코드"가 연구 PoC에선 더 평가받는다.**
