# M2 노트 — M1 대비 변화 + 어디까지 깊게 볼지

> 목적: M2(proxy_v2)는 "제품 수준 + 보안회사 제출" 기준으로 짠 것.
> 하지만 내 진짜 목표는 **파일 업로드 탐지(M6~M7)**.
> 그 관점에서 "꼭 이해할 것 / 그냥 넘어갈 것(오버 구현)"을 구분해 둔다.

---

## 1. M1 → M2 무엇이 달라졌나

| 항목 | M1 (proxy_v1) | M2 (proxy_v2) |
|---|---|---|
| 기본 구조 | blind relay (내용 안 보고 바이트만 퍼나름) | **파싱 루프** (헤드 파싱 → 프레이밍대로 body 전달 → 반복) |
| 핵심 도구 | 없음 | **SockReader** (버퍼드 리더) |
| 메시지 경계 | 모름 (그냥 흐름) | 안다 (Content-Length / chunked / until-close) |
| keep-alive | ❌ 요청 1개 처리하고 끝 | ✅ 한 연결에서 여러 요청 |
| 헤더 | `strstr("Host:")` 단순 | 맵 형태, **대소문자 무시** 파싱 |
| 보안 | ❌ | 헤더 크기 상한 / CL 오버플로우 / 스머글링 거부 / recv 타임아웃 |

### 왜 blind relay 를 버렸나 (가장 중요한 구조 변화)
- M1 의 `Relay` 는 바이트만 퍼나르니까 **"요청이 어디서 끝나는지"를 모른다.**
- 경계를 몰라서 **keep-alive(다음 요청 이어받기)가 불가능**하다.
- 그래서 M2 는 "한 메시지의 경계를 파악 → 다음으로" 구조로 바꿈.
- (blind relay 는 버린 게 아니라 **M3 CONNECT 터널 전용으로 보류** — 암호화 바이트는 파싱 못 하니 그땐 다시 blind 가 맞음)

## 2. ★ 꼭 이해하고 갈 메인 로직 (파일 업로드 탐지의 직접 토대)

이 3개는 **M6~M7에서 내가 직접 만질 부분**이라 제대로 이해해야 함.

### (A) 부분수신 누적 — `SockReader`
- TCP 는 스트림이라 recv 한 번에 메시지가 딱 안 떨어진다.
- 헤더가 쪼개져 오거나, 반대로 body 까지 미리 읽혀버린다(over-read).
- `SockReader` 가 "줄 단위 읽기 / N바이트 정확히 읽기 + 잉여 바이트 보관"을 해결.
- **파일은 무조건 크고 여러 recv 로 쪼개져 오므로 이 누적 개념이 필수.**

### (B) 헤더 파싱 — `readHead`, `headerGet`
- 업로드 식별의 핵심 단서가 헤더에 있다:
  - `Content-Type: multipart/form-data; boundary=...`
  - `Content-Length` (파일 크기 추정)
- M6 에서 이 헤더들을 보고 "업로드인지" 판단할 것.

### (C) body 프레이밍 — `forwardExact` / `forwardChunked` / `responseBodyMode`
- **여기가 M7 의 진짜 핵심.** 파일 바이너리는 body 안에 있다.
- body 가 어디서 시작해 어디서 끝나는지를:
  - `Content-Length` → 그만큼 정확히
  - `chunked` → 청크 단위로
  - 길이 모름 → 연결 종료까지
- M7 에서는 이 body 를 "그냥 전달"이 아니라 **multipart 경계로 잘라 파일 바이너리를 추출**하게 됨.
  지금 M2 의 body 전달 로직이 그 직전 단계.

## 3. 🟡 오버 구현된 부분 (지금은 "이런 게 있구나" 수준으로 넘어감)

"제품/보안 제출"이라 넣었지만, **파일 업로드 탐지 학습과는 거리가 있는** 것들.
동작 원리만 알고, 코드를 깊이 파지는 않기로 함.

| 부분 | 위치(함수) | 왜 오버인가 | 어느 수준까지 |
|---|---|---|---|
| keep-alive 루프 | `HandleClient` while + `wantsClose` | 탐지엔 필수 아님 (연결 1개당 요청 1개여도 탐지는 됨) | "한 연결로 여러 요청 받을 수 있게 한 거" 정도만 |
| request smuggling 방어 | `requestBodyMode` 의 CL+TE 거부 | 보안 가산점이지 업로드 탐지와 무관 | "CL 과 TE 동시면 위험해서 막는다" 개념만 |
| until-close 응답 처리 | `forwardUntilClose` | 드문 엣지케이스 | 거의 안 봐도 됨 |
| upstream 재사용 | `HandleClient` 의 target 비교 | 성능 최적화 | 거의 안 봐도 됨 |
| recv 타임아웃 / SO_REUSEADDR | `setRecvTimeout`, `RunServer` | 운영 안정성 | "멈춘 연결 정리용" 정도만 |

---

## 4. 한 줄 정리

- **M2 를 만든 것 자체는 오버 아님** (body 프레이밍은 M7 토대라 꼭 필요).
- 하지만 **M2 의 모든 부분을 똑같은 깊이로 공부하는 건 오버.**
- → **(A)(B)(C) 메인 로직만 제대로**, 🟡 부분은 개념만 훑고 통과.
- 책임님 말대로 시간은 **M6~M7(서비스별 업로드 포맷 역해석)** 에 몰 것.
