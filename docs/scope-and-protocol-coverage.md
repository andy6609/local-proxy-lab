## 0. 코드의 성격 (범위의 대전제)

- 이 코드는 **회사에 제출·제품화되는 코드가 아니다.** 파일 업로드 식별을 검증하는 **연구용 PoC**다.
- 그러나 **학습용 최소 코드도 아니다.** 나중에 **이력서/포트폴리오로 공개할 코드**라, **견고하게(solid)** 간다.
- **두 트랙을 구분한다:**

| 트랙 | 파일 | 성격 | 이 문서의 대상 |
|------|------|------|----------------|
| 학습용 | `proxy_MITM_pure/proxy_MITM_pure.cpp` | 개념 이해용 최소판. 별도 기능 안 얹음. **그대로 둔다.** | ✗ |
| **연구용(견고)** | `proxy_v4/proxy_v4.cpp` | 실제 식별 기능을 얹는 **본 트랙**. 프로토콜 커버리지 확장은 여기서. | ✔ |

- **견고함의 선 긋기:**
  - **유지(이력서 값어치):** v4 수준의 파서 견고성·에러처리·자원정리·로깅·문서화된 한계.
  - **지양(풀 프로덕트/과잉):** WFP 커널 통합, IOCP, fail-close 정책엔진, 분산 장비 구조.
- **식별(관측) vs 차단(enforcement):** 본 PoC의 목표선은 **업로드 식별 + Fiddler급 관측**까지다.
  `main_purpose.md`의 `allow`/`block`/`audit`·`403 Forbidden` 차단·L7 정책엔진(processRules/networkRules/L7 rules)은
  **제품 그림의 enforcement 층**이며, PoC에서는 "식별이 되면 정책은 그 위에 얹을 수 있다"까지만 보인다(차단 자체는 이번 목표 아님).
- **WFP:** 지금은 **제외**(시간). local 메인 로직이 견고히 돌고 나서의 다음 단계. 현재는 explicit proxy
  (`curl -x` / 브라우저 프록시 설정) 경로로 트래픽 유도. (`main_purpose.md` 의 WFP+transparent 구조는 **미래 그림**)

> **이력서 핵심 = "파일 업로드를 식별해 (회사) 제품에 기능으로 추가했다."** TLS MITM·프로토콜 처리는 그걸 가능케 한 수단.

## 1. 목표 프로토콜 커버리지 — 전체 그림

**웹페이지든 앱이든, 어떤 시스템이 올리는 파일이든 업로드를 식별**하려면, 실제 클라이언트가 쓰는
전송 프로토콜을 다 해석할 수 있어야 한다(= "Fiddler처럼 기능이 다 굴러간다"의 의미).

| 프로토콜 | 전송 | 상태 | 접근(단계) |
|----------|------|------|------------|
| **HTTP/1.1** | TCP+TLS | v4에서 됨 | 기존 파서 그대로 |
| **HTTP/2** | TCP+TLS | 미지원 | **Plan A**(ALPN로 1.1 강제) → **Plan B**(nghttp2로 실제 h2 파싱) |
| **HTTP/3 (QUIC)** | **UDP**+TLS1.3 | 미지원 | **Plan C** — 범위 안. 접근 방식 결정 필요(아래 §4) |

**핵심 관점:** h2·h3는 HTTP의 **의미(method/헤더/body)를 바꾼 게 아니라 전송·프레이밍만** 바꿨다.
프레이밍을 풀면 **같은 논리적 요청**이 나오고, 그 위의 **업로드 식별 로직은 공유**된다.
그래서 **식별 로직을 1.1에서 먼저 완성(Plan A)** 하고, h2/h3는 "앞단 디코더를 갈아끼워 같은 정보를 뽑는" 확장으로 붙인다.

## 2. HTTP/2 — Plan A → Plan B

### Plan A (지금): ALPN로 h2를 http/1.1로 눌러 앉히기
- **무엇:** 이미 맺은 h2를 끌어내리는 게 아니라, **TLS 핸드셰이크 ALPN 협상에서 h2가 선택되지 못하게** 막아 1.1로 시작시킴.
- **코드(v4) 변경 — 딱 두 지점:**
  1. 세션1(브라우저↔프록시) 서버 `SSL_CTX`에 `SSL_CTX_set_alpn_select_cb` → 무조건 `http/1.1` 선택
  2. 세션2(프록시↔서버) `ussl`에 `SSL_set_alpn_protos` → `http/1.1`만 광고
- **효과:** 양쪽 세션이 1.1로 서고 **기존 HTTP/1.1 파서가 전부 처리** → 복호화 평문을 Fiddler처럼 관측 + 업로드 식별 가능.
- **알려진 break(문서화할 한계):** **gRPC/h2-only 클라이언트**는 1.1 제안을 거부 → handshake 실패. (Plan B에서 해소)

### Plan B (다음): nghttp2로 실제 h2 파싱
- **왜 라이브러리:** h2는 바이너리 프레이밍 + **HPACK 헤더 압축(상태 있는 동적 테이블·허프만)** + **스트림 멀티플렉싱** + 흐름 제어.
  HPACK을 손으로 짜는 건 노동·리스크가 큼. mitmproxy·Fiddler도 라이브러리 사용.
- **해야 할 일:** ALPN을 반대로(h2 협상) → 협상 결과로 분기(h2/1.1 둘 다 지원) → 세션1(nghttp2 server)·세션2(nghttp2 client)를
  다리 놓는 **h2-to-h2 리버스 프록시** → `proxyExchange`의 선형 루프를 **stream별 멀티플렉싱 이벤트 루프**로 개조.
- **식별 관점:** `Content-Type`/`Content-Disposition`은 **HEADERS 프레임(HPACK)**, 파일 body는 **DATA 프레임(stream별 재조립)**.
  같은 정보, 추출 계층만 추가.

## 3. HTTP/1.1 경로에서 먼저 갖출 식별 커버리지 (이번 주~)

Plan A 위에서 "업로드 식별"이 실제로 되려면 1.1 경로에 다음이 필요(제품 하드닝이 아니라 **식별 커버리지**):
- `multipart/form-data` boundary 파싱 + part별 `Content-Disposition`(filename)·`Content-Type`
- **Content-Encoding(gzip/br/deflate) 압축 해제** — 안 풀면 body를 못 읽음
- chunked 재조립, keep-alive 요청/응답 경계
- **TCP 분할 시 단서 조각화** 처리 → "100% 식별 가능한가"(멘토 지적)의 실체
- **Fiddler급 가시화:** 요청/응답 헤더·상태·Content-* 를 실제로 보이게
- **대용량 body·streaming 처리** — 업로드 파일이 크면 전체 버퍼링이 메모리·지연 부담. **부분 검사(앞부분 N바이트로 매직넘버/헤더 식별)** 로 설계.
  (`main_purpose.md` §8.4의 "대용량 body·streaming은 별도 설계 필요"를 **식별 커버리지 항목으로 편입**)

## 4. HTTP/3 (QUIC) — 범위 안 (Plan C), 접근 방식 결정 필요

**QUIC/HTTP3는 이 프로젝트 범위 안이다.** 다만 TCP MITM으로는 손댈 수 없는 성격이라 접근을 정해야 한다.

> **실측(2026-07-14):** claude.ai에 파일 첨부 시 크롬 개발자도구 Network 탭에서 **`upload-file` 요청이 `h3`(QUIC)로 나감**을 확인.
> 즉 타깃 서비스(claude.ai)의 **업로드가 실제로 QUIC을 탄다** → 프록시가 QUIC을 처리(C-1 최소한 차단→폴백)하지 않으면 **업로드를 아예 못 본다**. C-1이 "있으면 좋은" 게 아니라 **필수 전제**임이 실측으로 확인됨. (측정: 프록시 없는 상태. 프록시 경유 시 크롬이 h2/1.1로 폴백하는지는 별도 재측정 필요.)

- **왜 다른가:** QUIC은 **UDP 위에서 도는 전송 프로토콜**이고 **TLS 1.3이 전송에 내장**돼 있다. HTTP/3가 그 위에 얹힘.
  → 현재 TCP 소켓 기반 MITM 경로로는 **아예 안 잡힘**(소켓 타입부터 다름).
- **두 갈래 접근:**

  | 접근 | 내용 | 노동 | 커버리지 |
  |------|------|------|----------|
  | **C-1 (권장 기본)** | **UDP:443(QUIC)을 차단** → 클라가 **TCP(h2/1.1)로 폴백** → 기존 MITM 경로로 흡수 | 작음 | "QUIC으로 새는 것 없음"은 달성. 단, 복호화가 아니라 폴백 유도 |
  | **C-2 (본격)** | **QUIC/HTTP3 자체를 MITM 복호화** — QUIC 라이브러리(ngtcp2/quiche/msquic) + **QPACK**(h3 헤더 압축) + HTTP/3 프레이밍 + UDP 처리 + QUIC TLS1.3에 동적 인증서 | **매우 큼**(h2보다 더) | QUIC 트래픽까지 실제 열람 |

- **현실적 순서:** **C-1(차단→폴백)** 을 먼저 넣어 "QUIC 우회 구멍"을 막고, **C-2(실제 QUIC 복호화)** 는
  h2(Plan B)까지 끝난 뒤의 **본격 확장**으로 둔다. 발표에서는 "QUIC은 왜 TCP MITM으로 안 되는지 + 우리가 택한 접근"을 설명.
- **[결정 필요]** QUIC을 **"막아서 폴백"(C-1)까지로 볼지, "실제 복호화"(C-2)까지 목표로 둘지** — 시간 예산이 크게 갈림. 멘토 확인 권장.

## 5. MCP 트래픽 관측 가능성 (열린 항목)

**배경:** 파일 유출 경로는 브라우저 업로드만이 아니다. AI 클라이언트(Claude Desktop, IDE 플러그인, 각종 CLI)가 **MCP(Model Context Protocol)** 서버를 통해 외부 도구·데이터에 접근하면, 그 경로로도 회사 데이터가 나갈 수 있다. 따라서 "업로드 식별"의 범위 논의에 MCP를 포함해 두되, **관측 가능 여부가 전송 방식에 따라 갈린다**는 점을 명확히 한다.

| 유형 | 관측 가능 여부 | 근거 |
|------|----------------|------|
| **HTTP/HTTPS 기반 MCP** (remote, SSE / streamable HTTP) | **조건부 가능** | TLS MITM 프록시가 CA 신뢰 하에 복호화 가능(일반 HTTPS 트래픽과 동일). 단 아래 [주의] 참조 |
| **stdio 기반 MCP** (local) | **불가능** (네트워크 계층 구조적 사각지대) | 소켓을 거치지 않는 로컬 IPC(stdin/stdout 파이프)이므로 관측 지점 자체가 없음. 보완하려면 별도의 호스트 기반(프로세스·파일) 모니터링이 필요 |

**[주의] "HTTP MCP는 무조건 가능"이 아니다.** HTTP 기반이라도 MCP 클라이언트가 브라우저가 아니라 별도 앱(Claude Desktop, IDE 플러그인 등)이면, 그 앱이 **시스템 루트 CA 신뢰 저장소를 사용하는지, 자체 인증서 피닝을 하는지**에 따라 캡처 가능 여부가 갈린다. 시스템 신뢰 저장소를 쓰면 우리 Root CA로 복호화되지만, 자체 피닝을 하면 4장의 피닝 사이트와 동일하게 MITM이 차단된다. 따라서 "조건부 가능"으로 표기한다.

### QUIC 한계와 stdio 한계는 층위가 다르다 (혼동 주의)

둘 다 겉보기엔 "우리 프록시로 못 잡는다"로 같아 보이지만, 원인과 보완 방향이 완전히 다르다.

| | QUIC(HTTP/3) 한계 | stdio MCP 한계 |
|---|---|---|
| **원인 층위** | 네트워크 계층에는 **존재**하나, UDP + 전송 내장 TLS 1.3 구조 탓에 TCP 기반 MITM이 개입하기 어려움 | 애초에 **네트워크 소켓을 안 씀**. 패킷도, host:port도 없음 |
| **성격** | "내 도구가 못 따라감" (도구의 한계) | "관측 지점 자체가 없음" (네트워크 모니터링 접근법의 사각지대) |
| **보완 방향** | 네트워크 레이어 안에서 해결 — QUIC 차단→TCP 폴백(§4 C-1) 같은 우회책 | 네트워크 계층을 **벗어나야** 해결 — ETW, 프로세스 생성·파일 핸들 후킹 등 **호스트 기반 모니터링**으로 접근 자체를 전환. 프록시를 아무리 개선해도 불가 |

즉 QUIC은 "여전히 네트워크 위의 문제"라 프록시 계열 해법으로 닿지만, stdio MCP는 "네트워크 밖의 문제"라 프록시 계열로는 원리적으로 닿지 않는다. 이 구분을 명시해 두어야, 향후 "stdio는 어떻게 보완하나"라는 질문에 "네트워크 프록시 개선"이 아니라 "호스트 기반 모니터링 도입"이라는 올바른 답으로 연결된다.

**[결정 필요]** MCP 감시를 이번 범위에 포함할지, 포함한다면 HTTP MCP 복호화까지인지 — 멘토 확인. (stdio는 이 프로젝트의 네트워크 기반 접근으로는 불가이므로, 필요 시 별도 트랙으로 분리)

## 6. 범위 밖 (지금 단계)

- **WFP 커널 통합 / transparent redirect** — 나중(local 완성 후).
- **풀 프로덕트 하드닝** — IOCP, fail-close 정책엔진, 분산 복호화 장비, 인증서 캐시 만료 정책 등.
- **인증서 피닝 사이트 복호화** — 원리적 불가(앱이 leaf fingerprint/public key pin을 자체 검증).
  증상: TLS handshake 실패·connection reset·`certificate verify failed`·`pinning validation failed`.
  MITM 불가 세션은 **raw tunnel로 통과(내용 못 봄)** 하고 **기록만** 하며, bypass/차단은 **운영 정책**으로(“맞다/틀리다보다 어떻게 운영하나”). (`main_purpose.md` §4.7/§8.2)
