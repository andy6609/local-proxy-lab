## 1. 목표 프로토콜 커버리지 — 전체 그림

**웹페이지든 앱이든, 어떤 시스템이 올리는 파일이든 업로드를 식별**하려면, 실제 클라이언트가 쓰는
전송 프로토콜을 다 해석할 수 있어야 한다(= "Fiddler처럼 기능이 다 굴러간다"의 의미).

| 프로토콜 | 전송 | 상태 | 접근(단계) |
|----------|------|------|------------|
| **HTTP/1.1** | TCP+TLS | v4에서 됨 | 기존 파서 그대로 |
| **HTTP/2** | TCP+TLS | 미지원 | **Plan A**(ALPN로 1.1 강제) → **Plan B**(nghttp2로 실제 h2 파싱) |
| **HTTP/3 (QUIC)** | **UDP**+TLS1.3 | 미지원 | **Plan C** — 범위 안. 접근 방식 결정 필요(아래 §4) |

**핵심 관점:** h2·h3는 HTTP의 **의미(method/헤더/body)를 바꾼 게 아니라 전송·프레이밍만** 바꿨다.
프레이밍을 풀면 **같은 논리적 요청**이 나오고, 그 위의 **업로드 식별 로직은 공유**된다.
그래서 **식별 로직을 1.1에서 먼저 완성(Plan A)** 하고, h2/h3는 "앞단 디코더를 갈아끼워 같은 정보를 뽑는" 확장으로 붙인다.

## 2. HTTP/2 — Plan A → Plan B

### Plan A (지금): ALPN로 h2를 http/1.1로 눌러 앉히기
- **무엇:** 이미 맺은 h2를 끌어내리는 게 아니라, **TLS 핸드셰이크 ALPN 협상에서 h2가 선택되지 못하게** 막아 1.1로 시작시킴.
- **코드(v4) 변경 — 딱 두 지점:**
  1. 세션1(브라우저↔프록시) 서버 `SSL_CTX`에 `SSL_CTX_set_alpn_select_cb` → 무조건 `http/1.1` 선택
  2. 세션2(프록시↔서버) `ussl`에 `SSL_set_alpn_protos` → `http/1.1`만 광고
- **효과:** 양쪽 세션이 1.1로 서고 **기존 HTTP/1.1 파서가 전부 처리** → 복호화 평문을 Fiddler처럼 관측 + 업로드 식별 가능.
- **알려진 break(문서화할 한계):** **gRPC/h2-only 클라이언트**는 1.1 제안을 거부 → handshake 실패. (Plan B에서 해소)

### Plan B (다음): nghttp2로 실제 h2 파싱
- **왜 라이브러리:** h2는 바이너리 프레이밍 + **HPACK 헤더 압축(상태 있는 동적 테이블·허프만)** + **스트림 멀티플렉싱** + 흐름 제어.
  HPACK을 손으로 짜는 건 노동·리스크가 큼. mitmproxy·Fiddler도 라이브러리 사용.
- **해야 할 일:** ALPN을 반대로(h2 협상) → 협상 결과로 분기(h2/1.1 둘 다 지원) → 세션1(nghttp2 server)·세션2(nghttp2 client)를
  다리 놓는 **h2-to-h2 리버스 프록시** → `proxyExchange`의 선형 루프를 **stream별 멀티플렉싱 이벤트 루프**로 개조.
- **식별 관점:** `Content-Type`/`Content-Disposition`은 **HEADERS 프레임(HPACK)**, 파일 body는 **DATA 프레임(stream별 재조립)**.
  같은 정보, 추출 계층만 추가.

## 3. HTTP/1.1 경로에서 먼저 갖출 식별 커버리지 (이번 주~)

Plan A 위에서 "업로드 식별"이 실제로 되려면 1.1 경로에 다음이 필요(제품 하드닝이 아니라 **식별 커버리지**):
- `multipart/form-data` boundary 파싱 + part별 `Content-Disposition`(filename)·`Content-Type`
- **Content-Encoding(gzip/br/deflate) 압축 해제** — 안 풀면 body를 못 읽음
- chunked 재조립, keep-alive 요청/응답 경계
- **TCP 분할 시 단서 조각화** 처리 → "100% 식별 가능한가"(멘토 지적)의 실체
- **Fiddler급 가시화:** 요청/응답 헤더·상태·Content-* 를 실제로 보이게

## 4. HTTP/3 (QUIC) — 범위 안 (Plan C), 접근 방식 결정 필요

**QUIC/HTTP3는 이 프로젝트 범위 안이다.** 다만 TCP MITM으로는 손댈 수 없는 성격이라 접근을 정해야 한다.

- **왜 다른가:** QUIC은 **UDP 위에서 도는 전송 프로토콜**이고 **TLS 1.3이 전송에 내장**돼 있다. HTTP/3가 그 위에 얹힘.
  → 현재 TCP 소켓 기반 MITM 경로로는 **아예 안 잡힘**(소켓 타입부터 다름).
- **두 갈래 접근:**

  | 접근 | 내용 | 노동 | 커버리지 |
  |------|------|------|----------|
  | **C-1 (권장 기본)** | **UDP:443(QUIC)을 차단** → 클라가 **TCP(h2/1.1)로 폴백** → 기존 MITM 경로로 흡수 | 작음 | "QUIC으로 새는 것 없음"은 달성. 단, 복호화가 아니라 폴백 유도 |
  | **C-2 (본격)** | **QUIC/HTTP3 자체를 MITM 복호화** — QUIC 라이브러리(ngtcp2/quiche/msquic) + **QPACK**(h3 헤더 압축) + HTTP/3 프레이밍 + UDP 처리 + QUIC TLS1.3에 동적 인증서 | **매우 큼**(h2보다 더) | QUIC 트래픽까지 실제 열람 |

- **현실적 순서:** **C-1(차단→폴백)** 을 먼저 넣어 "QUIC 우회 구멍"을 막고, **C-2(실제 QUIC 복호화)** 는
  h2(Plan B)까지 끝난 뒤의 **본격 확장**으로 둔다. 발표에서는 "QUIC은 왜 TCP MITM으로 안 되는지 + 우리가 택한 접근"을 설명.
- **[결정 필요]** QUIC을 **"막아서 폴백"(C-1)까지로 볼지, "실제 복호화"(C-2)까지 목표로 둘지** — 시간 예산이 크게 갈림. 멘토 확인 권장.

## 5. 범위 밖 (지금 단계)

- **WFP 커널 통합 / transparent redirect** — 나중(local 완성 후).
- **풀 프로덕트 하드닝** — IOCP, fail-close 정책엔진, 분산 복호화 장비, 인증서 캐시 만료 정책 등.
- **인증서 피닝 사이트 복호화** — 원리적 불가. 실패 세션 기록 + bypass/차단은 **운영 정책**으로(“맞다/틀리다보다 어떻게 운영하나”).
