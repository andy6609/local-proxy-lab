# 실행 계획 — 제품 수준 Local Proxy PoC (10주 커리큘럼 기반, WFP 제외)

> 공식 10주 커리큘럼(장동혁 책임님)을 **마일스톤 + 주차 매핑**으로 푼 실행 계획.
> 각 단계 = **동작하는 산출물 + 설명 한 줄(보고용) + 트레이드오프(학습용)**.
> (파일명은 `-to-mitm` 이지만, 실제 범위는 **W3~W10 = 프록시~파일탐지~보고서** 전부 포함)

---

## ⚠️ 전제 (꼭 기억)

1. **목표 = 제품 수준(production-grade) 실제 프록시.**
   - `proxy_relay` 등 실습 파일은 **개념 학습용**. 제품 코드 아님.
   - 실습의 `parseHost`(단순 `strstr`)는 **W4에서 견고한 파싱으로 재구현**(부분수신/헤더경계/keep-alive/방어).
2. **이 문서 범위 = Local Proxy ~ 파일 업로드 탐지. WFP(커널)는 "로컬 완성 후의 다음 단계"**(제외가 아니라 나중, 조건: 로컬 전부 완성 후). 이 단계 트래픽 유도는 **Explicit Proxy**(브라우저 프록시 설정 / `curl -x`).
3. **만드는 방식(책임님 조언 + 본인 의지 절충)** — 각 단계에 [직접]/[AI] 표시:
   - **[직접]** = 개념 핵심(중계·파싱·MITM 흐름·업로드 식별) → 손으로 짜며 이해
   - **[AI]** = 보일러플레이트/에러처리 골격/인증서·TLS API 호출부 → AI 활용 + "설명 가능"이면 됨

---

## 주차 ↔ 마일스톤 매핑

| 주차 | 커리큘럼 주제 | 마일스톤 |
|------|----------------|----------|
| W1~W2 | 개념 학습 (Proxy/HTTP/HTTPS/TLS/인증서, Explicit vs Transparent) | (학습 단계) |
| W3 | Local Proxy 기본 구현 (수락·연결·중계·종료) | **M1** 양방향 릴레이 + **M3** CONNECT 터널 |
| W4 | HTTP 요청/응답 분석 | **M2** 견고한 HTTP 파싱·로깅 |
| W5 | TLS MITM 구조·환경 구성 | **M4** MITM 기초(Root CA) |
| W6 | HTTPS 트래픽 분석 | **M5** MITM 완성(SNI 동적 인증서·평문 분석) |
| W7 | 파일 업로드 행위 식별 | **M6** 업로드 식별 |
| W8 | 파일 메타·바이너리 추출 | **M7** 추출·검증 |
| W9 | 파일 분석 + PoC 검증 | **M8** 분석·다환경 테스트 |
| W10 | 최종 보고서·발표 | **M9** 문서화·발표 |

> 현재 위치: **M0 완료(학습용 relay)** → 다음 **M1**.

---

## M0 — 현재: 학습용 turn-based relay  ✅ (제품 아님)
- 프록시 본질(받기→목적지→연결→전달→응답) 개념 검증 완료(curl로 httpbin 중계).
- 제품과의 차이(앞으로 메울 것): 단방향 순차, strstr 파싱, keep-alive·부분수신·에러처리·HTTPS 없음.

---

## M1 — 양방향(full-duplex) 릴레이  · W3
- **[목표]** 한 연결에서 양방향 동시 중계, 한쪽 끊기면 양쪽 정리.
- **[만들 것]** [직접] `relay(from,to)` + 동시화(스레드2/`WSAPoll`), 연결 생명주기 / [AI] 자원누수 방지·로깅 골격
- **[설명·시연]** "업로드/응답 방향을 동시에 중계, 종료 시 자원 회수."
- **[트레이드오프]** 스레드-per-방향 vs select(단순 vs 확장), 연결당 비용
- **[완료기준]** 큰 응답+keep-alive에서 데드락/누수 없이 양방향, 한쪽 종료 시 반대쪽도 종료.

## M2 — 견고한 HTTP 파싱 + 로깅  · W4
- **[목표]** Method/URL/Host/Header/Content-Type/Body 정확·안전 파싱 + 로그.
- **[만들 것]** [직접] 요청라인/헤더 파싱, **`\r\n\r\n` 경계**, **부분수신 누적**, `Content-Length`/**chunked** body, keep-alive 루프 / [AI] 헤더맵 자료구조·이상입력 방어·로그 포맷
- **[설명·시연]** "헤더가 쪼개져 와도, keep-alive로 여러 요청이 와도 정확히 파싱·기록한다."
- **[트레이드오프]** 직접 파서 vs 라이브러리, body 전체버퍼 vs 스트리밍(대용량)
- **[완료기준]** 실제 브라우저/curl의 분할·keep-alive·chunked 요청 안정 파싱 + 로그.

## M3 — HTTPS 터널 (CONNECT, raw 통과)  · W3 연장
- **[목표]** HTTPS를 복호화 없이 일단 통과(MITM 전 단계).
- **[만들 것]** [직접] `CONNECT host:443` 처리 → `200 Connection Established` → 이후 암호 바이트 양방향 릴레이(M1 재사용) / [AI] CONNECT 파싱 골격
- **[설명·시연]** "HTTPS는 아직 내용은 못 보지만 프록시 경유로 정상 접속(터널)."
- **[트레이드오프]** raw 터널=안전·호환 좋음, 단 **내용 못 봄** → M4/M5 필요 근거.
- **[완료기준]** 브라우저 프록시 설정 후 https 사이트 정상 로드.

## M4 — TLS MITM 기초 (Root CA + 클라측 TLS 서버)  · W5
- **[목표]** 프록시가 클라에게 "서버 역할"로 TLS handshake.
- **[만들 것]** [이해] 두 세션 구조 코드화 / [AI] **Root CA 생성 + Windows 신뢰 저장소(CurrentUser Root) 등록**, **TLS 라이브러리 선택**(★결정: SChannel(윈도우 통합) vs OpenSSL(이식·예제)) / [직접/이해] 고정 인증서로 클라측 TLS 서버 PoC
- **[설명·시연]** "Root CA를 신뢰시키면 프록시가 클라에게 TLS 서버로 행세 가능. (Root CA 필요 이유 설명)"
- **[트레이드오프]** SChannel vs OpenSSL, 신뢰등록 범위(CurrentUser vs LocalMachine)
- **[완료기준]** 고정 leaf로 클라와 TLS handshake 성공, 인증서 신뢰됨.

## M5 — TLS MITM 완성 (SNI 동적 인증서 + 평문 분석)  · W6
- **[목표]** 임의 https 목적지 자동 MITM → 평문 HTTP 분석 → 재암호화 전달.
- **[만들 것]** [AI] ClientHello **SNI 추출** → host용 **leaf 인증서 동적 생성+캐시** / [직접/이해] 세션1 복호화 →(M2 파서)평문 분석→ 세션2 재암호화 upstream, 응답 역방향 / [AI] 실패 시 raw fallback
- **[설명·시연]** "https의 URL/Header/Body를 평문으로 확인. 핀닝 등 실패는 fallback/기록."
- **[트레이드오프]** 인증서 생성 비용→캐시, **pinning/별도 trust store→MITM 실패(불가피)**, HTTP/2·QUIC·streaming 범위 밖
- **[완료기준]** 핀닝 없는 대표 https에서 평문 요청/응답 확인 + 재암호화 정상 전달, 실패 로깅.

## M6 — 파일 업로드 행위 식별  · W7  ★본 과제 핵심
- **[목표]** 업로드 요청을 L7 단서로 식별.
- **[만들 것]** [직접] `multipart/form-data` 판별, **`Content-Disposition` 파싱**(filename), `Content-Type`, 파일 크기 추정 / [관찰 먼저] 타깃 AI 서비스(ChatGPT/Gemini/네이버)가 실제로 어떻게 올리는지 캡처 → **서비스별 업로드 지문 표** 작성 / [AI] 멀티파트 파서 골격
- **[설명·시연]** "이 요청이 파일 업로드인지, 어떤 파일인지(이름/타입/크기) 식별 가능. 서비스별로 패턴이 이렇게 다르다."
- **[트레이드오프]** 서비스마다 방식 상이(multipart vs JSON+base64 vs resumable), 정규식/휴리스틱 한계
- **[완료기준]** 타깃 2~3개 서비스의 업로드를 식별, 지문 표 정리.

## M7 — 파일 메타 + 바이너리 추출  · W8
- **[목표]** 업로드에서 파일명/확장자/MIME/크기 + **바이너리** 추출, 원본 일치 검증.
- **[만들 것]** [직접] 멀티파트 본문에서 파일 part 경계 분리 → binary 추출 / [직접] 해시 비교(원본 vs 추출본) / [AI] 경계처리·대용량 스트리밍 골격
- **[설명·시연]** "업로드된 파일 바이너리를 뽑아 저장하고, 원본과 해시가 일치함을 검증."
- **[트레이드오프]** 메모리(대용량 buffering) vs 스트리밍, 부분 추출 정책
- **[완료기준]** 추출 파일이 원본과 바이트/해시 일치.

## M8 — 파일 분석 + PoC 검증  · W9
- **[목표]** 추출 바이너리로 파일유형/확장자/시그니처 분석 + 다환경 테스트.
- **[만들 것]** [직접] **매직넘버(시그니처)** 확인, 확장자 vs 실제유형 불일치 탐지 / [직접] 여러 브라우저/AI Agent에서 테스트, **성공/실패 정리** / [AI] 시그니처 DB·리포트 골격
- **[설명·시연]** "확장자 위조 탐지 가능. 서비스/환경별로 식별되는 것과 안 되는 것(경계)을 표로 제시."
- **[트레이드오프]** 시그니처 커버리지, false positive/negative, 핀닝/암호화로 불가한 케이스
- **[완료기준]** 다환경 테스트 결과표(성공/실패/원인) 완성.

## M9 — 최종 보고서 + 발표  · W10
- **[목표]** 목적/구조/구현/테스트결과/한계/개선방향 정리 + 발표.
- **[만들 것]** [직접] 보고서 서사·구조 / [AI] 다이어그램·문구 다듬기
- **[설명·시연]** "전체 프로젝트를 처음부터 끝까지 설명할 수 있다." ← 책임님 1순위 요구
- **[완료기준]** 보고서 + 발표자료 완성, 한계와 개선점 명확.

## 가로지르는 제품 요건 (모든 단계 공통)
- 에러처리/복원력(모든 소켓·TLS 결과 확인, 실패 시 안전 종료) · 자원 누수 0
- 로깅/관측(세션·결정·실패 기록 → 후에 Viewer/보고서로) · 설정화(포트·타깃·MITM on/off·bypass)
- 동시성/성능(다수 연결 안정성)

## 진행 원칙
1. 한 마일스톤 = 동작 산출물 + 설명 한 줄 + 트레이드오프 메모.
2. M1~M3는 [직접] 비중 높게. M4~M5는 흐름 이해+AI 골격+설명 가능. M6~M8은 [직접] 다시 높게(본 과제 핵심).
3. 매 단계 실제 실행 검증 + 안 되는 케이스(한계)도 기록.
4. 한 번에 다 만들지 않고 점진적으로 → 트레이드오프 체감.
