# Step 3: 호스트 파싱 + 양방향 릴레이 (무엇이 바뀌고 얼마나 어려운가)

> 계획상 step 3 = **"호스트 파싱 + 양방향 릴레이(중계)"**
> 브라우저 요청에서 `Host:` 목적지를 뽑아 → 진짜 서버에 `connect` → 데이터를 양쪽으로 중계.

---

## 1. "서버"에서 "중간 다리"로

### server_clitent_proxy.c 에서는 

```
브라우저 → [내 프로그램] : 받아서 출력하고 끝
```

`HandleClient`는 소켓 하나(`clientSock`, 브라우저와 연결)만 다룸.

### 이제는 내가 정말 프록시가 된다

```
브라우저 → [내 프록시] → 진짜 서버(httpbin.org 등)
브라우저 ← [내 프록시] ← 진짜 서버
```

**내가 양쪽 사이에 낀 다리**가 된다.
`HandleClient` 안에서 **소켓을 하나 더** 만든다:

- `clientSock` : 브라우저와 연결된 소켓 (기존)
- `upstream`   : 진짜 서버와 연결되는 소켓 (**추가**)

그리고 둘 사이로 바이트를 퍼나른다(= 릴레이).

---

## 2. 새로 필요한 4가지

| 단계 | 새 개념/함수 | 난이도 |
|---|---|---|
| ① Host 파싱 | `strstr`/`strtok`로 `Host:` 줄에서 목적지 뽑기 | ⭐⭐ 쉬움~보통 |
| ② 도메인→IP 변환 | `getaddrinfo` (DNS 조회) — `"httpbin.org"`는 문자열이라 바로 connect 못 함 | ⭐⭐ 보통 (새 API) |
| ③ 업스트림 연결 | `socket` + `connect` (이미 아는 것!) | ⭐ 쉬움 |
| ④ 릴레이(중계) | 받은 바이트를 받은 만큼 그대로 `send`, **루프로** | ⭐⭐⭐ 보통~어려움 |

## 3. 수도코드 — 먼저 "쉬운 버전"부터

처음엔 양방향을 다 하지 말고, **"요청 한 번 전달 → 응답 되돌려주기"**만 하는 단순 버전이 좋다.
(GET 요청엔 이걸로 충분하고, 브라우저에 실제 페이지가 뜬다.)

1. 브라우저 -> 요청을 한 번 받음
2. 그 요청을 진짜 서버에 그대로 전달
3. 진짜 서버의 응답을 다 받아서 브라우저에 돌려줌
4. 끝, 연결 닫음 

=> 무전기처럼 "한 명이 말 끝내면 -> 상대가 답한다". 동시에 말 못함, 한 방향 끝나야 다음 방향

언제 충분한가: 단순 GET 요청: "주소창에 치고 -> 페이지 받기" 처럼 묻고 답받는게 명확히 나뉘는 경우. 
한계: 요청과 응답이 동시에 오가는 상황을 못 다룸. 큰 파일을 올리는 동시에 서버가 진행 상황을 내려보내거나, 한 연결로 여러 요청이 오가는 keep alive 경우, 실시간 통시니 안됨

```c
void HandleClient(SOCKET clientSock) {
    // 1. 브라우저 요청 받기
    char buffer[8192];
    int n = recv(clientSock, buffer, sizeof(buffer), 0);
    if (n <= 0) { closesocket(clientSock); return; }

    // 2. 요청 헤더에서 Host 뽑기 (헤더는 텍스트라 파싱 OK)
    //    "Host: httpbin.org\r\n"  ->  "httpbin.org"
    char host[256];
    parseHost(buffer, host);     // strstr("Host:") 찾고 \r\n 전까지 복사
    int port = 80;               // http 기본 포트

    // 3. 도메인 이름 -> IP 로 변환 (DNS 조회)
    //    문자열 "httpbin.org" 는 connect 에 바로 못 씀
    struct addrinfo* server = resolve(host, port);   // getaddrinfo 래핑

    // 4. 진짜 서버로 가는 '새 소켓' 만들고 연결
    SOCKET upstream = socket(AF_INET, SOCK_STREAM, IPPROTO_TCP);
    connect(upstream, server->ai_addr, server->ai_addrlen);

    // 5. 브라우저가 보낸 요청을 그대로 진짜 서버에 전달
    send(upstream, buffer, n, 0);    // ★ n바이트(받은 만큼) 그대로! strlen 금지

    // 6. 진짜 서버의 응답을 받아 브라우저로 되돌려줌 (릴레이 루프)
    while ((n = recv(upstream, buffer, sizeof(buffer), 0)) > 0) {
        send(clientSock, buffer, n, 0);   // ★ 받은 만큼 그대로 (응답은 바이너리)
    }

    // 7. 양쪽 다 닫기
    closesocket(upstream);
    closesocket(clientSock);
}
```

→ 이 버전이면 **브라우저로 진짜 웹페이지가 뜬다.** "내 프록시가 중계했다"가 눈에 보인다.

## 4. 그 다음 "진짜 양방향" 업그레이드 (난이도 ↑)

위 버전은 "요청 보내고 → 응답 받기"가 **순서대로**다.
하지만 실제 프록시(특히 최종 목표인 **파일 업로드 = 큰 POST**, 또는 HTTPS 터널)는
**양쪽이 동시에** 데이터를 주고받아야 한다.
방향이 2개라 **한 방향은 별도 스레드로** 돌린다:

```c
// 양방향 동시 릴레이
void relay(SOCKET from, SOCKET to) {
    char buf[8192]; int n;
    while ((n = recv(from, buf, sizeof(buf), 0)) > 0)
        send(to, buf, n, 0);
    shutdown(to, SD_SEND);   // 한쪽 끊기면 반대쪽도 끝나도록 신호
}

void HandleClient(SOCKET clientSock) {
    // ... Host 파싱 + upstream 연결까지 동일 ...

    std::thread t(relay, clientSock, upstream); // 방향1: 브라우저 -> 서버
    relay(upstream, clientSock);                // 방향2: 서버 -> 브라우저 (현재 스레드)
    t.join();

    closesocket(upstream);
    closesocket(clientSock);
}
```

여기가 step 3의 진짜 난관:
**"한쪽 연결이 끊기면 반대쪽도 같이 정리"** 하는 자원 관리.
`shutdown`/`closesocket` 타이밍이 꼬이기 쉽다. (계획 노트에 적은 그 포인트)
: 데이터가 언제 어느 방향에서 올지 예측 불가다. 만약 "브라우저에서 받기"만 기다리고 있으면, 그동안 서버가 보낸 데이터는 처리 못 하고 막혀버림(반대도 마찬가지). 그래서 두 방향을 동시에 감시해야 하는데, 방법이: 
한 방향은 별도 스레드에 맡기고, 다른 방향은 현재 스레드에서 돌림 -> 두 흐름이 동시에 굴러감
(또는 select 로 두 소켓을 한꺼번에 감시 - 더 고급)

한쪽이 끊겼을 때 반대쪽도 같이 정리하는 거임. 전화로 치면 한 명이 끊었는데 다른 쪽이 계속 "여보세요" 하고 있으면 안되니까/

그러니까 한 방향은 별도 스레드에게 맡기고, 다른 방향은 현재 스레드에서 돌리는건데 -> 그럼 두 흐름이 동시에 흘러감
(또는 select 으로 두 소켓을 한꺼번에 감시 - 고급)

-> 이게 필요한 이유: 최종 목표가 파일 업로드 탐지이고, 업로드는 브라우저가 서버로 큰 데이터를 계속 밀어 올리는 거기 떄문에 양방향이 동시에 흐르는 전형적 케이스임.

실제 케이스들 

뭐 쉬운 예시로 큰 파일을 업로드하는 중인데, 서버가 다 받기도 전에 답을 줌
예: 파일이 너무 크면 서버가 업로드 도중에 "413 Too Large(너무 큼)" 또는 "401 권한 없음"을 보내버림. 브라우저는 아직 올리는 중인데 서버는 이미 "그만 보내, 거부야"라고 내려보내는거임. → 올라가는 차선 + 내려오는 차선이 동시에 활성.

② HTTP "100 Continue" (실제 프로토콜 기능)
브라우저가 큰 파일 올리기 전에 "이거 보내도 돼?"라고 헤더만 먼저 보냄. 그러면 서버가 본문을 받기도 전에 "100 Continue, 보내도 좋아" 라고 먼저 답함. → 요청 본문 보내는 도중에 서버가 말을 거는 상황.

③ WebSocket / 실시간 스트리밍 (ChatGPT가 딱 이거!)
ChatGPT 답변이 글자가 주르륵 흘러나오는 거는 그건 한 연결을 열어두고 서버가 계속 밀어내는 방식임. 그동안 사용자가 "중단" 버튼을 누르면 브라우저가 동시에 서버로 신호를 보냄. → 양방향 동시.

④ HTTPS 터널 (CONNECT)
프록시가 암호화된 바이트를 양쪽으로 그냥 흘려보내는 모드. 안의 내용을 모르니 어느 방향에서 언제 올지 전혀 예측 못 함 → 무조건 양방향 대기

## 왜 프록시는 양방향이어야 하나

프록시는 중간에 낀 입장이라 "지금 요청 단계인지 응답 단계인지" 정확히 모를때가 많음
그래서:
        "올라오는 것도 받고, 내려오는 것도 받을 준비를 동시에 해줌"
예시 1번 처럼 거부 메세지 같은거를 놓치고 멈출 수 있기 때문에.

## 정리
TCP 는 2차선 (올라가는 선/ 내려오는선 )이 독립이라 동시에 다닐 수 있음. full- duplex
단순 GET은 턴제처럼 보여서 OK 이지만
업로드 중 서버가 답하는 경우 "100 continute, chat gpt streaming, http 터널 같이 동시 통신이 있어서 양방향 필요


### 진행 순서 

1. 먼저 **Host 파싱**만 떼서 연습 (받은 buffer에서 `Host:` 뽑아 printf)
2. **쉬운 버전**(요청 전달 + 응답 릴레이) 완성 → 브라우저에 페이지 뜨는 거 확인
3. 그 다음 **양방향**으로 확장

step 1→2 했던 것처럼 단계적으로.

## 핵심 요약

> step 3 = **서버 → 중간 다리(프록시)** 로 역할 전환.
> 소켓이 1개(`clientSock`) → 2개(`clientSock` + `upstream`)로 늘고,
> 새 함수는 **Host 파싱 / getaddrinfo(DNS) / connect / 릴레이 루프**.
> **쉬운 버전(순차) → 양방향(동시)** 순서로 가는 게 안전.
> 릴레이는 항상 **받은 바이트 수만큼 그대로 send** (응답은 바이너리라 `%s`/`strlen` 금지 — [socket-buffer-and-string.md](socket-buffer-and-string.md) 참고).
