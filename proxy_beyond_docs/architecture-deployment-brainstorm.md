# 배치 구조 브레인스토밍 — 어디서 복호화하고 프록시를 어디에 둘 것인가

> 팀 공유용. **아래는 전부 확정이 아니라 브레인스토밍/열린 옵션**이다. 멘토 확인 후 방향을 정한다.
> 현재 코드는 "로컬 프록시" 방식으로 시작한 상태.

---

## 1. 무엇을 정하는 문제인가

TLS 복호화(MITM)를 **어디서** 수행하고, 대상 트래픽을 **어떻게** 프록시로 유도하느냐의 배치(deployment) 구조 문제다. 탐지 로직 자체와는 별개의 축이다.

---

## 2. 복호화를 어디서 하나 — 두 방향

| 방식 | 내용 | 성격 |
|------|------|------|
| **로컬 프록시** (현재) | 각 PC에서 프록시가 직접 복호화 | 지금 코드. 검증·PoC에 적합 |
| **중앙 복호화 장비/서버** | 트래픽을 한 곳에 모아 복호화 | 기업 보안 제품의 일반적 형태. 규모/운영에 유리 |

- 현재는 **로컬 프록시**로 시작한다.
- 중앙 장비 방식은 브레인스토밍에서 나온 미래 옵션이며, 지금 범위는 아니다.

---

## 3. 트래픽을 어떻게 프록시로 유도하나 — "각 클라이언트마다"의 문제

브라우저는 프록시 설정을 따르지만, **AI를 브라우저로만 쓰는 것이 아니다.** CLI 도구나 데스크톱 앱(예: 각종 AI CLI, 데스크톱 클라이언트)으로도 사용하며, 이들은 브라우저 프록시 설정을 따르지 않을 수 있다.

- 그래서 "각 클라이언트(CLI/앱)마다 그 트래픽을 프록시로 태우도록 보장해야 한다"는 문제가 생긴다.
- 이 문제의 근본 해결책이 **WFP(투명 프록시)** 다. 앱이 프록시 설정을 무시해도 OS 레벨에서 트래픽을 프록시로 우회시킨다.
- 단, WFP는 **지금 범위 밖**(로컬 메인 로직 견고화 후의 다음 단계). 현재는 explicit proxy(브라우저 프록시 설정 / `curl -x`)로 유도한다.

---

## 4. MCP 감시 — 열린 항목

- MCP(Model Context Protocol) 경로, 즉 AI가 외부 도구·데이터에 접근하는 경로도 감시 대상에 포함할지가 논의에서 나왔다.
- **맥락이 더 필요하며 확정된 바 없다.** 멘토 재확인 항목.

---

## 5. 정리 — 현재 결정 상태

| 항목 | 상태 |
|------|------|
| 복호화 위치: 로컬 프록시 | 현재 채택 (PoC) |
| 복호화 위치: 중앙 장비/서버 | 브레인스토밍, 미래 옵션 |
| 클라이언트 유도: explicit proxy | 현재 채택 |
| 클라이언트 유도: WFP(투명) | 범위 밖, 다음 단계 |
| MCP 경로 감시 | 열린 항목, 멘토 확인 |

> 요약: **지금은 "로컬 프록시 + explicit proxy"로 간다.** 중앙 장비·WFP·MCP 감시는 방향이 정해지지 않은 확장 후보이며, 멘토 확인 후 로드맵에 반영한다.
